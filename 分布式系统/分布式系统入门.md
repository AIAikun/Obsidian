分布式的核心目标：用多台机器解决 “单台机器性能 / 存储不够” 的问题

## **数据 / 请求的分布逻辑**（数据分片）

#### **哈希分片**

假设有这么一种场景：我们有三台缓存服务器分别为：node0、node1、node2，有3000万个缓存数据需要存储在这三台服务器组成的集群中，希望可以将这些数据均匀的缓存到三台机器上，你会想到什么方案呢？

我们可能首先想到的方案是：取模算法hash（key）%N，即：对缓存数据的key进行hash运算后取模，N是机器的数量；运算后的结果映射对应集群中的节点。具体如下图所示：

问题：
假设现在有个缓存 key，它的 hash 运算结果是 **10**：

- 当服务器数量 N=3 时：10%3=1 → 这个 key 的缓存会存在 node1 上。
- 当服务器扩容到 N=4 时：10%4=2 → 系统会去 node2 上找这个 key 的缓存，但实际上这个缓存还在原来的 node1 上。

这时候，系统 “找不到” 这个已有的缓存，就会认为 “这个缓存不存在”（也就是**缓存失效**），只能去数据库重新加载数据。

如果是 3000 万数据，服务器数量一变，几乎所有旧缓存的 “存储位置” 都会和新计算的位置不匹配，相当于**所有旧缓存同时失效**—— 这时候所有请求都会涌向数据库，数据库扛不住就会崩溃，也就是 “缓存雪崩”

#### **一致性哈希算法**

1. **先建一个 “哈希环”**
    
    把哈希值的范围（比如 0 到 2³²-1）想象成一个**环形（首尾相连）**，这就是一致性哈希的 “基础空间”。
    
2. **把服务器节点映射到环上**
    
    对每台服务器的标识（比如 IP、名称）做哈希运算，得到一个哈希值，然后把这个值 “放” 到哈希环的对应位置上。
    
    比如：
    

- node0 的哈希值算出来是 100 → 环上位置 100
- node1 的哈希值是 200 → 环上位置 200
- node2 的哈希值是 300 → 环上位置 300

3. **数据怎么映射到服务器？**
    
    对缓存 key 做哈希运算，得到一个哈希值，也放到环上，然后**顺时针找最近的服务器节点**，这个节点就是该数据的存储节点。
    
    比如：

- key1 的哈希值是 150 → 顺时针最近的是 node1（位置 200）→ 存在 node1
- key2 的哈希值是 250 → 顺时针最近的是 node2（位置 300）→ 存在 node2

#### 为什么能解决 “扩容 / 缩容导致缓存雪崩”？

当服务器数量变化时，**只有部分数据的存储节点会改变**，不是所有数据都失效：

比如原来有 node0、node1、node2，现在扩容加一个 node3（哈希值 250，环上位置 250）：

- 原本哈希值在 200~300 之间的数据，之前是存在 node2；现在哈希值 200~250 的会改存在 node3，250~300 的还存在 node2。
- 其他范围（比如 100~200、300~100）的数据，存储节点完全不变。

这样扩容后，只有 “node1 到 node3 之间” 的小部分数据需要重新分配，大部分旧缓存依然有效，不会出现 “所有缓存同时失效” 的雪崩问题。
